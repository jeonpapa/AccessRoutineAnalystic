# 멀티에이전트 모니터링 시스템 — 아키텍처 패턴

> **목적**: 수집·모니터링·감시 성격의 멀티에이전트 시스템을 설계할 때 재사용 가능한 도메인 중립 패턴 모음.
> MA_AI_Dossier(약가·허가 모니터링) 프로젝트에서 추출. 다른 프로젝트에 그대로 복사해 적용 가능.
> 도메인 특화 로직(특정 API·계산식·스키마)은 제거하고 **구조 패턴**만 남김.

---

## 0. 이 패턴이 맞는 프로젝트

- 외부 소스(웹사이트·공공 API·뉴스)에서 데이터를 **주기적으로 수집**
- 수집 데이터를 **가공·분석**하여 사용자에게 제공
- "어제와 무엇이 달라졌는가"를 **추적·감시**
- 여러 에이전트가 협업하며, 결과 품질을 **자동 검증**해야 함

맞지 않는 경우: 단발성 스크립트, 실시간 트랜잭션 처리, 순수 CRUD 앱.

---

## 1. 에이전트 4그룹 분류

모든 에이전트를 역할로 4그룹에 배치한다. 새 에이전트를 만들 때 "이건 어느 그룹인가"를 먼저 정한다.

| 그룹 | 역할 | 특성 | 예시 |
|---|---|---|---|
| **수집 (Collector)** | 외부 소스 → DB | 스크레이퍼/API 클라이언트. 캐시 TTL 보유. 실패 허용(빈 결과 명시) | 사이트 스크레이퍼, 공공 API 클라이언트, 뉴스 수집기 |
| **가공 (Processor)** | 수집 데이터 → 분석 결과 | LLM/규칙 엔진 활용. 입력은 DB, 출력도 DB 또는 캐시 | 원인 분석기, 집계기, 가중치 보정기 |
| **감시 (Guard)** | 결과·코드·룰 검증 | 다른 에이전트의 산출물을 입력으로 받음. 회귀·편차 탐지 | 코드 회귀 감시, 합의↔런타임 감사, 결과 다수결 검증 |
| **제어/산출 (Control/Output)** | 분배·렌더링 | 요청 분배(orchestrator), 최종 산출물 생성(dashboard/API) | 오케스트레이터, 대쉬보드 생성기, API 서버 |

**원칙**: 한 에이전트는 한 그룹에만 속한다. 수집기가 가공까지 하면 분리한다.

---

## 2. 단방향 데이터 흐름 (핵심 원칙)

```
[수집]  외부 소스 → 스크레이퍼/API 클라이언트 → DB
[가공]  DB → 가공 에이전트 → DB (또는 캐시)
[산출]  DB → API/대쉬보드 (읽기 전용)
```

**규칙**:
- 산출 계층(대쉬보드/API)은 **DB만 읽는다**. 역방향 쓰기 금지
- 수집 → 가공 → 산출은 단방향. 순환 의존 금지
- 각 단계는 DB 를 경유해 다음 단계로 전달 (직접 호출 최소화)

**왜**: 단계 간 결합도를 낮춰 각 에이전트를 독립 테스트·재실행 가능하게 한다. 스크레이퍼가 실패해도 대쉬보드는 이전 DB 상태로 동작.

## 3. 세 가지 실행 플로우

시스템은 세 가지 진입점을 가진다. 새 기능이 어느 플로우에 속하는지 먼저 정한다.

### A. 스케줄러 자동 (배치)
정기 cron 으로 수집 → 가공 → 감시를 순차 실행.
```
00:00  백업
02-04  수집 (외부 소스 → DB)
05:00  감시 1 — 합의/룰 ↔ 런타임 감사
06:00  감시 2 — 코드 회귀 + 개선 제안
09:00  산출 — 대쉬보드 갱신
```
**핵심**: 수집이 끝난 직후 감시를 배치한다. 데이터가 갱신되면 곧바로 검증되도록.

### B. API 실시간 (사용자 조회)
사용자 요청 시 즉시 응답. 캐시 우선, miss 시에만 수집.
```
사용자 검색 → 캐시 hit? → DB 반환
                ↓ miss
            수집 에이전트 실행 → DB 저장 → 반환
```

### C. 대화 기반 (오케스트레이터)
복잡한 요청을 분석해 작업 계획(work plan)을 세우고 에이전트를 순차 실행.
```
요청 → 오케스트레이터(룰 대조 + 영향도 분석) → work plan
     → 에이전트 순차 실행 → 감시 에이전트 최종 검증
```

## 4. 2중 감시 루프 (가장 이식 가치 높은 패턴)

감시를 **두 축**으로 분리한다. 하나만으로는 사각지대가 생긴다.

### 축 1 — 코드/회귀 감시 (Quality Guard)
- 코드베이스에서 **금지 패턴** 스캔 (하드코딩, 위험한 추론 로직 등)
- **baseline 회귀 검증** — 알려진 정답 N개를 매일 재계산해 일치 확인
- 큰 파일·미해결 편차 등 **개선 제안** 생성
- 출력: `review_YYYY-MM-DD.md` (사람이 읽는 리포트)

### 축 2 — 합의↔런타임 감사 (Rule Compliance)
- 사용자와 **합의한 규칙**(메모리/결정 기록)이 실제 런타임에서 지켜지는지 감사
- 각 합의 항목마다 **증거 수집 함수**를 레지스트리에 등록 — "룰이 있다"가 아니라 "증거가 이렇다"
- 검증 불가능한 항목은 **명시적 SKIP** (묵시적 통과 금지)
- 출력: `compliance_YYYY-MM-DD.md` (PASS/FAIL/SKIP)

**왜 둘로 나누나**: 축 1은 *코드가 규칙을 어겼는가*, 축 2는 *합의가 코드에 반영됐는가*. 코드는 깨끗한데 합의가 누락될 수 있고, 그 반대도 가능. 두 축이 교차 검증.

### 합의 감사 구현 패턴
```
합의 기록(메모리 .md) ──→ 인덱스 파싱 ──→ 각 항목마다:
                                          check 함수 있나?
                                          ├─ 있음 → 실행 → PASS/FAIL + 수치
                                          └─ 없음 → SKIP + 사유 명시
```
신규 합의 추가 시 **check 함수 또는 SKIP 사유를 반드시 등록**하게 강제. 안 하면 리포트에 "미검증 신규 항목" 경고.

## 5. 수집 에이전트 공통 패턴

### 5.1 베이스 클래스
모든 수집기는 공통 베이스를 상속. 인터페이스 통일:
```
BaseCollector
  - search(query) → list[Record]      # 필수 구현
  - 캐시 read/write (TTL 보유)
  - 빈 결과 sentinel 저장              # 재조회 폭주 방지
```

### 5.2 캐시 TTL 분리
- **성공 결과**: 긴 TTL (예: 30~90일)
- **실패/빈 결과**: 짧은 TTL 또는 sentinel — 다음 주기에 재시도
- 실패를 성공과 같은 TTL 로 캐시하면 "한 번 실패 → 30일간 빈 값" 버그 발생

### 5.3 변형 검색 fallback
외부 소스가 정확매칭만 지원할 때, 검색어 변형을 우선순위대로 시도하고 첫 hit 에서 중단:
```
1. 원본 검색어
2. 표기 변형 (공백/하이픈/대소문자/언어별 표기)
3. prefix 축약
4. 상위 카테고리 검색
첫 hit → 중단
```

### 5.4 출처/추정 구분 필수
수집 데이터에 **출처 라벨**을 항상 붙인다:
- `source = official_api | scraped | estimated | inherited`
- 추정값과 실측값을 DB 컬럼/필드로 구분. UI 가 신뢰도를 표시할 수 있게

## 6. 가공 에이전트 패턴

### 6.1 다단계 fallback
1차 소스가 실패하면 2차로 자동 강등:
```
1차: 고품질 소스 (정확하지만 느림/비쌈)
  ↓ 실패
2차: 대체 소스 (빠르지만 거침)
  ↓ 실패
3차: 휴리스틱 기본값 (명시적 'unverified' 라벨)
```

### 6.2 LLM 출력은 검증 계층을 거친다
LLM 이 생성한 결과는 곧바로 신뢰하지 않는다:
- 규칙 엔진으로 형식·범위 검증
- 다수결(아래 §7)로 교차 검증
- 사용자 확정이 필요한 영역은 admin UI 로 분리

---

## 7. 결과 검증 — 다수결 (Review)

LLM 판단이 애매한 영역은 단일 모델 결정 대신 **독립 검증자 다수결**:
```
결과 + 원래 요청 → 검증자 A (모델 1) ─┐
                  → 검증자 B (모델 2) ─┼→ 점수 합산 → approve / reject
                  → 기계적 규칙 체크   ─┘
```
- 검증 4축: ① 요청 부합 ② 룰 준수 ③ 근거 품질 ④ 일관성
- 검증자 모델 키가 없으면 자동 비활성 — 단일 검증으로 graceful degrade

## 8. 멀티에이전트 공용 컨텍스트

여러 AI 도구(Claude Code · Codex · Gemini · IDE 확장 등)가 같은 프로젝트를 작업할 때:

### 8.1 단일 진입점 — `AGENTS.md`
프로젝트 루트에 `AGENTS.md` 하나. Codex 가 자동 로드하고, 다른 도구도 root markdown 으로 인식. 내용:
- 환경/자격증명 위치
- 작업 컨텍스트 어디서 읽나 (룰·메모리·plan·보고서 경로)
- 에이전트 아키텍처 한 줄 요약
- 규칙 맵 (상세 룰 파일 인덱스)
- 절대 금지 / 최소 원칙
- 신규 에이전트 온보딩 체크리스트

### 8.2 룰 분산 — `rules/` 인덱스 패턴
상세 룰은 영역별 파일(`rules/<영역>_rules.md`)로 분산. `AGENTS.md` 와 `CLAUDE.md` 는 **인덱스만** 유지. 룰이 한 파일에 몰리면 비대해지고 충돌 감지가 어려움.

### 8.3 도구 전용 메모리 노출 — symlink
특정 도구(예: Claude Code)의 메모리/plan 이 도구 전용 경로(`~/.claude/...`)에 있으면, 프로젝트 안에 `.agent-context/` 디렉토리를 만들고 symlink 로 노출:
```
.agent-context/
  memory/ → (도구 전용 메모리 경로)
  plans/  → (도구 전용 plan 경로)
```
- 다른 도구도 프로젝트 폴더 안에서 읽을 수 있음
- `.gitignore` 에 `.agent-context/` 등록 — 개인 메모리가 repo 에 commit 되지 않게

## 9. 새 프로젝트에 이식하는 체크리스트

1. **그룹 정하기** — 만들 에이전트를 4그룹(수집/가공/감시/산출)에 배치
2. **DB 를 경유점으로** — 단방향 흐름. 산출 계층은 읽기 전용
3. **수집기 베이스 클래스** — `search()` 인터페이스 + 캐시 TTL(성공/실패 분리)
4. **2중 감시 루프** — 코드 회귀 감시 + 합의↔런타임 감사. 둘 다 매일 자동 실행
5. **합의 감사 레지스트리** — 합의 항목마다 check 함수 또는 SKIP 사유 등록 강제
6. **스케줄러 배치 순서** — 수집 직후 감시. `백업 → 수집 → 감시 → 산출`
7. **출처 라벨** — 모든 수집 데이터에 official/scraped/estimated 구분
8. **LLM 결과는 다수결** — 애매한 판단은 독립 검증자 2+ 로 교차 검증
9. **공용 컨텍스트** — `AGENTS.md` 단일 진입점 + `rules/` 인덱스 + 도구 메모리 symlink
10. **회귀 baseline** — 알려진 정답 N개를 감시 에이전트가 매일 재검증

## 10. 안티패턴 (하지 말 것)

| 안티패턴 | 왜 나쁜가 | 대신 |
|---|---|---|
| 수집기가 가공까지 수행 | 재실행·테스트 단위가 커짐 | 그룹 분리 |
| 산출 계층이 DB 에 직접 쓰기 | 단방향 흐름 붕괴, 데이터 출처 불명 | 읽기 전용 |
| 실패 결과를 성공과 같은 TTL 캐시 | 한 번 실패 → 장기간 빈 값 | 실패 TTL 분리 |
| 감시 1축만 운영 | 코드는 깨끗한데 합의 누락 (또는 반대) 사각지대 | 2중 감시 |
| 합의 추가 후 check 함수 미등록 | 묵시적 통과 — 지켜지는지 모름 | 등록 강제 + SKIP 명시 |
| LLM 출력 곧바로 신뢰 | 할루시네이션이 DB 에 영구 저장 | 검증 계층 + 다수결 |
| 추정값을 실측값과 동일 필드 | UI 가 신뢰도 구분 불가 | 출처 라벨 컬럼 |
| 룰을 한 파일에 전부 | 비대화, 충돌 감지 불가 | rules/ 인덱스 분산 |

---

*이 문서는 구조 패턴만 담는다. 구체 구현(특정 API·스키마·계산식)은 각 프로젝트의 도메인 룰 문서에 둔다.*
